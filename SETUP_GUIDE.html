Setup guide: automated backend dev blog + portfolio

Everything below is free. Total setup time: roughly 2–3 hours if you follow it in order.

Stack decision, upfront: Astro (site) + GitHub Pages (hosting) + Google Gemini API free tier (writing) + LinkedIn's free Consumer API tier (posting) + GitHub Actions (scheduling). No credit card, no server to maintain, nothing that expires. NVIDIA NIM and self-hosted open-source models were considered — NIM's free tier is credit-based rather than perpetual, and self-hosting a model means running a GPU box, which defeats "free and zero-maintenance" for a once-a-week job. Gemini's Flash tier gives you far more requests than you'll ever use for one post a week, forever, with no card on file. Oracle Cloud's Always Free tier is genuinely lifetime-free, but you don't need a server for a static site — GitHub Pages already gives you free hosting with zero maintenance. Keep Oracle in your back pocket if you ever want to self-host something stateful, like an analytics dashboard.
Steps
  1. Get the code
  2. Push to a new GitHub repo
  3. Point your domain at GitHub Pages
  4. Turn on GitHub Pages
  5. Get a free Gemini API key
  6. Create a LinkedIn app and get a refresh token
  7. Add all the GitHub secrets
  8. Personalize the site
  9. Test everything end to end
  10. SEO submission (one-time)
  11. Ongoing: writing your own posts, adjusting the schedule

1Get the code

You were given a ready-to-use project (project.zip) alongside this guide. Unzip it — you'll see:

project/
├── site/                  the Astro portfolio + blog
├── automation/             the weekly post generator + LinkedIn poster
├── .github/workflows/      the two GitHub Actions that tie it together
└── README.md

Everything already builds — it was tested before being handed to you. You don't need to run npm create astro or scaffold anything; just personalize it (step 8) and push it.

2Push to a new GitHub repo

  1. Create a new empty repository on GitHub (no README, no .gitignore — you already have those). Call it whatever you like, e.g. bishalregmi-site.
  2. From inside the unzipped project folder: 
    git init
git add .
git commit -m "initial commit"
git branch -M main
git remote add origin https://github.com/YOUR_USERNAME/YOUR_REPO.git
git push -u origin main

3Point your domain at GitHub Pages

You already own bishalregmi.com.np. At your domain registrar's DNS settings, add:

TypeName/HostValue
CNAMEwww (or leave root, see note)YOUR_USERNAME.github.io

If you want the bare domain (bishalregmi.com.np with no www) to work, GitHub's docs will have you add 4 A records instead, pointing to GitHub's IPs (185.199.108.153, .109.153, .110.153, .111.153). Add a file site/public/CNAME containing just your domain name — this tells GitHub Pages which custom domain to serve.

echo "bishalregmi.com.np" > site/public/CNAME
git add site/public/CNAME && git commit -m "add custom domain" && git push

DNS can take anywhere from a few minutes to a few hours to propagate — this is a good step to do first thing Saturday morning so it's ready by the time you finish everything else.

4Turn on GitHub Pages

  1. In your repo: Settings → Pages.
  2. Under "Build and deployment", set Source to GitHub Actions (not "Deploy from a branch" — the included deploy.yml workflow handles the build itself).
  3. Under "Custom domain", enter bishalregmi.com.np and save. Once DNS resolves, check "Enforce HTTPS".
  4. Push anything to main (or just re-push from step 2) and watch the Actions tab — the "Build and deploy site" workflow should run and go green in about a minute.

5Get a free Gemini API key

  1. Go to aistudio.google.com/app/apikey and sign in with any Google account.
  2. Click "Create API key". No credit card required.
  3. Copy the key — you'll paste it into GitHub secrets in step 7. The gemini-2.5-flash model this project uses has a free daily quota that's an order of magnitude more than the one request per week you actually need.

6Create a LinkedIn app and get a refresh token

This is the fiddliest step — LinkedIn's setup has more clicks than Gemini's, but it's still free and doesn't require anyone's approval for posting to your own profile.

6a. Create the app

  1. Go to linkedin.com/developers/apps → "Create app".
  2. Fill in an app name (e.g. "Blog Publisher"), and — this is required even though you're only posting to your own profile — link it to a LinkedIn Company Page. If you don't have one, create a free placeholder page for yourself first (Settings → Create a Company Page), then select it here.
  3. Upload any square logo, agree to the terms, click "Create app".

6b. Add the right products

  1. On your new app's page, go to the Products tab.
  2. Request "Sign In with LinkedIn using OpenID Connect" — this is instant, no review.
  3. Request "Share on LinkedIn" — also self-serve for posting to your own profile.

6c. Configure OAuth

  1. Go to the Auth tab. Note your Client ID and Client Secret.
  2. Under "Authorized redirect URLs for your app", add exactly: http://localhost:8787/callback

6d. Run the one-time local auth script

This has to run on your own machine (not GitHub Actions) because it opens a browser tab for you to approve access once.

cd automation
npm install
export LINKEDIN_CLIENT_ID="paste your client id"
export LINKEDIN_CLIENT_SECRET="paste your client secret"
npm run linkedin-auth

Local .env file

Instead of exporting variables each time, create a local .env at the project root (this file is gitignored). Copy .env.example and fill the values for local testing:

# .env (local, do NOT commit)
GEMINI_API_KEY=your_gemini_api_key_here
LINKEDIN_CLIENT_ID=your_linkedin_client_id_here
LINKEDIN_CLIENT_SECRET=your_linkedin_client_secret_here
# The refresh token and person URN are produced by the auth script and should be saved as GitHub Secrets
# LINKEDIN_REFRESH_TOKEN=...
# LINKEDIN_PERSON_URN=urn:li:person:...

Load the local .env into your shell before running the auth script:

set -a && source .env && set +a
cd automation
npm run linkedin-auth

A browser tab opens, asks you to approve, then redirects back to localhost. Your terminal will print three values — copy all of them somewhere safe:

LINKEDIN_REFRESH_TOKEN = ...
LINKEDIN_PERSON_URN = urn:li:person:...
The refresh token is valid for 365 days. Once a year, re-run this same script to get a fresh one and update the GitHub secret. Everything in between happens automatically — the weekly workflow exchanges this refresh token for a new 60-day access token on every run, so you never have to touch it week to week.

7Add all the GitHub secrets

In your repo: Settings → Secrets and variables → Actions → New repository secret. Add these six:

Secret nameWhere it came from
GEMINI_API_KEYStep 5
LINKEDIN_CLIENT_IDStep 6c
LINKEDIN_CLIENT_SECRETStep 6c
LINKEDIN_REFRESH_TOKENStep 6d output
LINKEDIN_PERSON_URNStep 6d output

8Personalize the site

Everything you need to edit lives in site/src/pages/index.astro:

  • Replace YOUR_GITHUB and YOUR_LINKEDIN with your real profile URLs (appears twice — hero sameAs block and the contact links).
  • Replace you@example.com with your real email.
  • Edit the techStack array to match what you actually use.
  • Rewrite the one-line tagline under your name to sound like you, not me.

Also drop a real og-image.png (1200×630px works well) into site/public/ — this is the image shown when your links are shared on LinkedIn/Twitter. A simple image with your name and title on a solid background is enough.

Commit and push — the deploy workflow rebuilds automatically.

9Test everything end to end

  1. In your repo's Actions tab, click "Weekly AI blog post" → "Run workflow" → confirm. Don't wait for Monday to find out something's misconfigured.
  2. Watch the run. It should: pick a topic, call Gemini, write a new markdown file, commit it, push, then post to LinkedIn.
  3. If the LinkedIn step fails, it's almost always one of: a typo'd secret, the refresh token not being saved correctly, or the app missing the "Share on LinkedIn" product from step 6b — the error message from LinkedIn's API will usually say which.
  4. Once it succeeds, the push to main automatically triggers "Build and deploy site" — check that one goes green too, then visit your site and confirm the new post is live.

10SEO submission (one-time)

  • Go to Google Search Console, add your domain as a property, verify via the DNS TXT record method, and submit https://bishalregmi.com.np/sitemap-index.xml.
  • Go to Bing Webmaster Tools and do the same — Bing lets you import directly from Search Console in one click.
  • Confirm https://bishalregmi.com.np/robots.txt and https://bishalregmi.com.np/rss.xml both load once the site is live.

Everything else — meta tags, Open Graph tags, canonical URLs, per-post JSON-LD BlogPosting schema, a Person schema on the homepage, and a dedicated indexable page per tag — is already wired into the templates. You don't need to do anything per-post; it's automatic for every post, AI-written or not.

11Ongoing: your own posts + adjusting things

Writing a post yourself, any time

Add a markdown file to site/src/content/blog/ with frontmatter matching this shape, then commit and push:

---
title: "Your title"
description: "Under 155 characters, shows up in search results"
pubDate: 2026-07-10
tags: ["postgresql", "backend"]
category: "postgresql"
draft: false
aiAssisted: false
readingTime: 6
---

Your content here in markdown.

Publishing the AI post on demand instead of waiting for Monday

Actions tab → "Weekly AI blog post" → "Run workflow". This is the exact same button you used for testing in step 9 — use it any time you want an extra post between the weekly ones.

Changing the schedule

Edit the cron line in .github/workflows/weekly-post.yml. It's currently 0 9 * * 1 (Mondays, 09:00 UTC) — crontab.guru is the easiest way to build a different one.

Adding more topics

automation/content-bank/topics.json holds 81 topics across 7 categories, pulled from your original blueprint. Add more objects to the topics array any time — same shape: title, category, focus. The script automatically avoids repeating a topic until the whole list has been used once, tracked in automation/content-bank/used-topics.json.

Skipping Medium

Deliberately left out — Medium stopped issuing new API integration tokens, so there's no free way to automate posting there for a new account. If you want to cross-post a specific piece manually, it's copy-paste, a couple of minutes.
